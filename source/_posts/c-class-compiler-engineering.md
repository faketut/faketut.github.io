这篇文章总结一个 **WLP4（类 C 教学语言）编译器后端** 的优化过程：输入是带类型信息的语法树，输出是 **AArch64 汇编**。目标并不只是“把语义翻译对”，而是把生成器优化到：**常见表达式少做事、函数调用少保存、栈帧布局更紧凑、指针运算更直接、生成代码更稳定**。

---

## 1. 系统目标与核心约束

### 1.1 这类编译器最重要的三件事
- **语义正确**：WLP4 的整数、指针、过程调用、堆分配、条件分支都必须严格按语言规则落到汇编。
- **输出高效**：同样的 AST，能少发几条指令、少访几次内存、少做几次栈调整，运行体感会明显不同。
- **实现可维护**：优化不能把代码生成器变成“补丁堆”，否则后续再加特性（如更多表达式模式）会很痛苦。

### 1.2 现实约束
- **前端已经固定**：输入不是自由文本，而是按产生式展开、附带类型后缀的树形中间表示，后端要顺着这套结构工作。
- **目标架构有 ABI 约束**：AArch64 要求调用点 `sp` 保持 16 字节对齐，参数寄存器、返回寄存器、保存寄存器都有规则。
- **WLP4 不是“只算整数”**：它既有 `long`，也有 `long*`，所以加减比较都要带类型意识；尤其是指针偏移，不能偷懒。

---

## 2. 总体架构：一个轻量 AST + 一个有状态代码生成器

这份实现的结构非常清晰：

- **Parser**：逐行读取前端输出，构造带 `kind / rhs / children / type` 的 AST。
- **Node 缓存层**：节点上缓存 `memoConst`、`memoPure`、`memoSu`、`memoHash`，为后续分析留钩子。
- **CodeGen**：维护符号表、变量类型、寄存器分配、当前过程的栈帧信息、字面量池、标签生成器。
- **过程级生成**：先收集过程、扫描 feature usage，再按过程逐个发射汇编，最后按需补 `getchar` / `putchar` stub。

这个设计的关键点是：**把“语义判断”和“指令发射”尽量局部化**。  
比如“这个表达式是不是常量”“这个乘零能不能删”“这个指针偏移能不能直接寻址”，都在离表达式最近的位置完成，不需要引入完整 IR 才能做优化。

## 3. 性能指标设计：先明确“后端快”体现在哪里

做编译器后端优化时，不能只盯着“编译成功”。至少要看 4 类成本：

### 3.1 指令条数
- 常量表达式能不能在编译期折叠掉？
- `x + 0`、`x * 1`、`x - x` 这类模式能不能直接省掉？
- 指针访问能不能直接生成一次 load/store，而不是先算地址再取值？

### 3.2 内存访问次数
- 局部变量是否必须每次都从栈上读？
- 调用前是否把所有临时值都压栈了？
- 栈帧里是否存在重复保存 / 恢复？

### 3.3 调用开销
- 每次过程调用是否都保存了不必要的寄存器？
- 参数多于 8 个时是否正确、且尽量紧凑地走栈传参？
- prologue/epilogue 是否有多余的 `sp` 调整？

### 3.4 实现复杂度
- 优化是局部可控的，还是已经把生成器变成到处耦合？
- 新特性能否沿现有模式扩展，而不是整体重写？

---

## 4. 第一层优化：把“能在编译期知道的事”尽量提前知道

### 4.1 常量折叠：最稳定、收益最高的优化之一

这份实现里 `isConst()` 会沿 AST 递归判断表达式能否在编译期求值，覆盖了：

- 数字字面量
- `NULL`
- 括号表达式
- `+ - * / %`
- 一部分代数恒等式

这类优化的价值不是“炫技”，而是**直接减少运行期指令**。例如：

- `x + 0 => x`
- `x * 1 => x`
- `0 * pure_expr => 0`
- `ptr + 3 => ptr + 24`

尤其是在教学语言这类表达式比较规整的场景里，**常量折叠的命中率很高，且几乎没有副作用风险**。

### 4.2 纯表达式判断：不是所有“乘 0”都能删

这里一个很工程化的点是：实现没有粗暴地做代数化简，而是先通过 `isPure()` 判断子树是否无副作用。

例如：

- `0 * (a + b)` 可以折叠为 0
- `0 * getchar()` 就不能删，因为 `getchar()` 有副作用
- `0 * new int[n]` 也不能直接删，因为堆分配本身有语义

这体现了编译器优化里一个非常重要的原则：**等价变换必须以语义保真为前提**。  
很多“看起来显然”的优化，一旦碰到函数调用、I/O、内存分配，就不再显然。

---

## 5. 栈帧优化：把调用约定真正落成“单一、稳定、可推理”的布局

### 5.1 单一栈帧模型，比到处 push/pop 更稳

这份代码在注释里明确写了一个高影响优化：**Single frame**。

核心思路是：

- 过程入口一次性分配完整 frame
- `x29` 指向统一定义的位置（参数区起点）
- 局部变量、返回地址、保存寄存器、scratch 区都在固定相对位置
- 退出时一次性恢复

这样做的好处非常直接：

- **偏移更容易推导**
- **表达式求值时少了零碎的栈操作**
- **debug 更容易**
- **后续做寄存器驻留/临时值保存时不容易把栈布局搞乱**

### 5.2 16 字节对齐不是“为了规范”，而是为了别踩 ABI 雷

AArch64 调用点要求 `sp` 16-byte aligned。  
这份实现通过 `align16Up()` 统一处理：

- frame 总大小对齐
- 调用时保存临时寄存器的空间对齐
- 超过 8 个参数时的栈参数区也对齐

如果不把这个约束收敛成公共函数，后面每加一个 call-site 特判，代码都会更脆。  
所以这里的优化不仅是性能优化，也是**正确性和可维护性优化**。

### 5.3 只有偏移小的时候直接 `ldur/stur`

WLP4 的局部变量访问本质上都是 frame-relative。  
但 AArch64 对带立即数偏移的寻址范围有限，所以实现里分成两档：

- **小偏移**：直接 `ldur/stur [x29, imm]`
- **大偏移**：先把 offset 装到临时寄存器，再 `add` 出地址后访问

这类“按编码能力分层”的做法很实用：  
**常见情况走快路径，极端情况保正确性**，而不是统一用最保守、最慢的方式。

## 6. 字面量与常量加载优化：减少重复常量、减少无意义搬运

### 6.1 过程级字面量池：把重复常量收敛起来

实现中最值得写进总结的一点，是 **literal pool**：

- 每个过程单独维护 `payloadToId`
- `emitLoadLitPayload()` 先记录 fixup
- `finalizeLiteralPool()` 在过程末尾统一补 `.8byte`
- 相同 payload 去重

这个优化的意义有三层：

1. **重复常量不重复发**
2. **常量地址装载逻辑统一**
3. **过程内部代码尺寸更稳定，便于后续继续做局部优化**

相比“哪里要常量就原地插一段 literal”，这种设计工程上干净得多。

### 6.2 `0` 不走常规常量加载路径

`emitLoadConst()` 对 0 做了特判，直接用：

- `emitZeroReg(reg)`  

本质上是通过寄存器自减清零，而不是去走字面量池或额外搬运。

这看起来只是一个小技巧，但在编译器后端里这类小技巧会频繁出现。  
因为 `0` 是极高频常量，任何对它的优化都会被大量放大。

### 6.3 “mov” 也做成显式语义，而不是依赖伪指令

这里没有完全信任汇编器的伪指令兼容性，而是用：

- 先清零目标寄存器
- 再 `add dst, dst, src`

来实现寄存器移动。

这类处理说明作者考虑了**工具链兼容性**。  
编译器工程不是只在理想汇编器上跑；一旦 grader / 教学环境 / linkasm 有限制，保守但可预期的输出通常更值钱。

---

## 7. 表达式生成优化：让常见模式直接命中快路径

### 7.1 乘法和加法的代数特例，收益非常高

在 `genTerm()` / `genExpr()` 里可以看到大量“先判模式，再走快路径”的逻辑：

- `x * 1 => x`
- `1 * x => x`
- `x * 0 => 0`（前提是另一边 pure）
- `x * 2 => x + x`
- `x + 0 => x`
- `0 + x => x`
- `x - x => 0`

为什么这种优化特别适合 WLP4？

- 语法树结构简单，匹配成本低
- 没有复杂别名分析需求
- 表达式深度通常不算特别夸张
- 运行环境往往对额外指令数很敏感

换句话说，这是那种**实现成本很低，但能持续吃到真实收益**的优化。

### 7.2 指针运算特化：不要把“元素偏移”当普通整数加减

WLP4 里 `long* + long`、`long* - long`、`long* - long*` 的语义都不同。  
这份实现把类型判断直接带进表达式生成：

- 指针加整数：整数先乘元素大小（这里是 8）
- 指针减整数：同理减 `8 * offset`
- 指针减指针：先做地址差，再除以 8

如果这一步偷懒，编译器“表面上能跑”，但语义就已经错了。  
而一旦把类型系统真正接到代码生成器里，很多优化也会自然出现，比如常量偏移可直接乘成字节数。

### 7.3 指针偏移识别：把 `*(p + 3)` 变成直接寻址

`isPtrOffset()` 是这份实现里很关键的一个局部优化点。  
它识别出：

- `ptr + const`
- `const + ptr`
- `ptr - const`

并把逻辑偏移转成字节偏移，如果落在目标架构支持的寻址范围内，就能在 load/store 时直接生成：

- `ldur/ldr [base, imm]`
- `stur/str [base, imm]`

这会省掉：

1. 一个显式地址计算
2. 一个额外临时寄存器
3. 若干无必要的中间结果保存

这种优化对数组访问、指针遍历特别有价值，因为这些模式在编译器作业和基准里通常非常高频。

## 8. 调用点优化：不是每次函数调用都值得“全量保存现场”

### 8.1 只保存会被 clobber、且当前真的在用的寄存器

`emitSaveTempsForCall()` 的策略很务实：

- 只看 `nextFree` 之前可能被当前表达式占用的寄存器
- 跳过目标寄存器
- 只保存必要的寄存器
- 保存区按 16-byte 对齐

这比“见 call 就把一堆寄存器全压栈”高效很多。  
对于表达式中嵌套调用、或参数本身又是表达式的情况，收益尤其明显。

### 8.2 调用后恢复也做了顺序优化

`emitRestoreTempsAfterCall()` 有一个很细的工程点：

- 先恢复非 `x0`
- 再把返回值挪到目标寄存器
- 最后如果之前保存了 `x0`，再恢复旧 `x0`

这避免了“返回值刚拿到就被覆盖”的问题，也让调用返回路径更稳定。  
这类小细节特别能区分“能过样例”和“真的考虑过寄存器生命周期”。

### 8.3 超过 8 个参数时，栈传参也保持结构化

AArch64 前 8 个参数走 `x0-x7`，更多参数走栈。  
实现里对超额参数：

- 先一次性分配对齐后的栈参数区
- 按顺序写入
- 调用结束后统一回收

这比边求值边 push 更适合作为编译器长期维护的模式，因为参数布局是**显式可推导**的。

---

## 9. 局部变量驻留寄存器：一个“小型寄存器分配器”的高性价比版本

### 9.1 不是所有局部变量都值得长期放寄存器

`buildSymbolTable()` 做了一个非常实用的决策：

- 扫描使用次数
- 排除被取地址的变量
- 按使用频率排序
- 选 top-k（最多 9 个）放到 `x19-x27`

这不是完整图着色寄存器分配，但在 WLP4 这种规模下，已经非常够用。

好处有两个：

1. **高频局部变量减少 frame load/store**
2. **实现复杂度远低于完整 RA**

### 9.2 “被取地址的变量不上寄存器”是个非常关键的保守条件

只要某个变量出现在 `&x` 里，它就必须有一个稳定内存位置；否则后续通过地址读写就会出错。  
所以实现把这类变量排除出 `regTab`，是非常正确的工程保守策略。

这说明一个成熟的后端优化不只是“尽量 aggressive”，而是**知道哪些地方必须收手**。

---

## 10. 控制流与运行时支持：把正确性问题尽量局部封装

### 10.1 条件跳转要区分有符号整数和指针比较

`genTest()` / `genInvertedTest()` 里一个容易被忽略但很重要的点是：

- `long` 比较走 `lt/le/ge/gt`
- 指针比较走无符号分支 `lo/ls/hs/hi`

这反映出：  
**类型系统不是只服务语义检查，它还直接影响最终选择哪条机器分支指令。**

### 10.2 运行时 stub 只在需要时引入

实现会先扫描 AST，看是否真的用到了：

- `println`
- `getchar`
- `putchar`
- `new / delete`

只有命中才发 `.import` 和对应 stub。  
这是一个典型的“代码体积优化 + 工程卫生优化”：输出不带多余运行时依赖，结果也更干净。

### 10.3 堆相关初始化只在 `wain` 且需要 heap 时做

`init` 只在需要堆、且入口是 `wain` 时调用；并根据首参是否是数组指针决定初始化参数。  
这说明后端并不是简单“看到 heap feature 就无脑初始化”，而是按语言入口语义做了约束。

## 11. 这份实现最值得保留的工程经验

### 11.1 不一定要引入 IR，局部 AST 优化也能做出不错效果

很多人一想到“编译器优化”，就默认要上三地址码、SSA、CFG 全家桶。  
但在 WLP4 这种目标明确、语言规模受控的项目里，**直接在 AST 上做局部常量折叠、纯度判断、模式匹配和调用点优化，已经能拿到很可观的收益**。

### 11.2 把 ABI 约束变成公共工具，而不是散落特判

像：

- 16 字节对齐
- frame-relative load/store
- 参数寄存器与栈参数
- 保存寄存器恢复顺序

这些都被收敛成了 `emit*` 辅助函数。  
这类封装的真正收益，不只是“代码好看”，而是后续继续加优化时**不容易把调用约定弄坏**。

### 11.3 优化顺序应该先抓“高频低风险”

如果要给这类项目排优先级，我会建议：

- **第一优先级**：常量折叠、代数化简、指针偏移特化
- **第二优先级**：单一栈帧、调用点最小保存、寄存器驻留
- **第三优先级**：字面量池去重、运行时按需引入、代码尺寸清理
- **长期项**：真正的表达式复杂度排序、活跃区间分析、尾调用优化落地

原因很简单：前两层通常最能直接改善指令数和内存访问，而实现风险仍可控。

---

## 12. 还可以继续优化的方向

这份实现已经有明显的“优化意识”，但如果继续往下做，还可以考虑：

- **真正用上 `memoSu` / Sethi-Ullman**：按表达式寄存器压力决定左右子树求值顺序，进一步减少 spill。
- **尾调用优化落地**：代码里已有 `isTailCall()` 钩子，但还没完全打通到最终发射路径。
- **更激进的常量传播**：跨 statement 的局部常量环境，可让一些 if/while 条件提前简化。
- **更细的 strength reduction**：例如乘以 4/8/16 转移位或地址模式。
- **过程级代码大小统计**：让优化不只是“感觉变快”，而是有数据支持。

---

## 13. 结语：教学编译器的优化价值，不在“卷复杂度”，而在“把关键路径做对”

WLP4 这类项目最容易陷入的误区，是把“能输出汇编”当作终点。  
但真正有工程味道的后端，会继续问：

- 这个表达式是不是本来就不用算？
- 这个局部变量是不是根本不该每次都读栈？
- 这个指针访问是不是可以直接命中寻址模式？
- 这个函数调用是不是保存了太多不需要保存的东西？

这份实现给出的答案是：**先把最常见、最确定、最容易保证语义正确的优化做好**。  
不需要一步到 SSA，也不需要为了“高级”而引入一堆中间层。只要抓住常量、栈帧、调用点、寄存器和类型驱动代码生成这几条主线，一个教学编译器就能从“会翻译”走到“有后端味道”。
