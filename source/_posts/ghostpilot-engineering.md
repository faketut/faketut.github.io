这篇文章总结一个 Windows 桌面端**实时面试辅助系统**（典型链路：**隐形悬浮窗** + **ASR（语音转文字）→ 文本大模型流式回答** + **截图热键 → 多模态大模型** + **本地 RAG 知识库**）在工程落地过程中的优化方法与踩坑经验。目标不是堆功能，而是把系统优化到：**听得快、答得快、不卡顿、不互相拖累、失败可自愈**。

---

## 1. 系统目标与核心约束

### 1.1 体验目标（最重要的三件事）
- **低端到端延迟**：用户说完一句话后，辅助答案要尽快开始“流式出现”。
- **高稳定性**：ASR/LLM 任一环节失败不能把整个应用拖死；异常要可恢复。
- **隐蔽与不打扰**：悬浮窗默认**点击穿透**，必要时可切换可拖拽/交互。

### 1.2 现实约束
- **Windows 音频采集复杂**：WASAPI loopback、设备切换、静音、采样率不一致都会带来不稳定。
- **云端依赖不可控**：ASR/LLM 受网络抖动与限流影响，需要工程层缓冲与降级。
- **多管线并行**：语音与截图两条链路必须隔离，避免“截图失败导致语音中断”。

---

## 2. 总体架构：把链路拆成两条“互不干扰”的管线

### 2.1 语音（ASR → 文本 LLM）主链路
- **WASAPI Loopback Capture**：采集系统输出（面试官语音通常来自会议软件扬声器）
- **ASR Streaming**：拿到 partial / final
- **Partial Segmenter**：把 partial 按标点/超时切句，避免 LLM 频繁被打断或长句才触发
- **RAGManager**：本地知识库召回（简历/项目亮点/常用模板）
- **Text LLM（OpenAI-compatible）**：以流式方式输出回答
- **ASR Overlay**：悬浮窗实时展示问答

### 2.2 截图（Alt+P → Vision LLM）独立链路
- **热键触发**：Alt+P
- **区域截图 + 压缩**：减少上传体积，稳定延迟
- **Vision LLM（如 Gemini）**：识别题目/代码/表格并输出
- **Vision Overlay**：独立显示，不阻塞语音主链路

> 架构优化的关键点：**把“可能慢、可能失败、可能卡 UI”的部分全部隔离**，让主链路始终可用。

---

## 3. 性能指标设计：先定义“快”到底是什么

### 3.1 建议定义 4 个关键延迟
- **T_capture→ASR_partial**：音频进入到拿到第一段 partial 的时间
- **T_final_sentence**：一句话完成（final）到触发 LLM 请求的时间
- **T_llm_first_token**：LLM 请求发出到第一个 token 出现的时间
- **T_e2e_first_answer**：用户停顿/句末到悬浮窗开始出现“可读回答”的时间

### 3.2 结果呈现要“可观测”
- 每次回答记录：分句触发原因（标点/超时/手动）、prompt 长度、RAG 命中文档、token 速率、重试次数。
- UI 上不需要暴露这些，但日志必须可查，否则优化没有抓手。

---

## 4. 语音链路优化：从输入到触发 LLM 的“关键路径”减法

### 4.1 音频采集的稳定性：静音与中断是常态
**问题**：会议软件切换输出设备、短暂静音、采集线程抖动，都会导致 ASR 断流或“听不见”。  
**实践**：
- **silence padding**：在采集端对静音段进行填充/对齐，避免 ASR 因无数据被动断开。
- **Audio Watchdog**：检测采集是否卡住，必要时自动重启采集与 ASR 会话（自愈比提示用户点按钮更重要）。
- **采样率/通道统一**：在本地做一次格式规范化，降低云端拒收或识别异常的概率。

### 4.2 Partial Segmenter：降低“触发 LLM”与“上下文错乱”的双重成本
**问题**：
- 直接用 ASR partial 触发会导致 LLM 频繁重入、回答抖动。
- 等 final 又会让响应显得慢，错过“先给一部分答案”的体验优势。  
**实践**：
- **标点触发**：识别到句号/问号/换行等，立即切句触发。
- **超时触发**：用户停顿达到阈值（如 800–1200ms）就切句，即便没有标点。
- **合并策略**：对短 partial 做合并；对重复片段做去重，避免“重复问句”污染 prompt。
- **速率限制**：同一时间只允许一个 LLM 会话占用“主回答槽位”，新句子进入队列或做轻量合并。

> 这个模块通常是端到端延迟优化的最大杠杆：它决定“什么时候值得问 LLM”。

## 5. LLM 侧优化：让答案“更快出现、更贴题、更稳定”

### 5.1 Prompt 预算：把 token 花在刀刃上
- **系统提示词短而强**：明确输出格式（要点式、可直接口述）、回答长度上限、避免长篇大论。
- **对话上下文裁剪**：只保留最近 N 轮 + 当前问题；历史只留摘要或关键实体。
- **RAG 注入控制**：只注入 top-k（如 3–5）且每条限制字数；避免把整个简历塞进去导致首 token 变慢。

### 5.2 RAG 工程：启动快、召回准、注入稳
- **本地知识库按文件夹约定**：如 `knowledge/` 放简历、项目笔记、常见答题模板。
- **chunk 策略**：按段落/标题切分，避免“一个 chunk 太大”导致召回冗余。
- **embedding 可选**：依赖不可用时（如无 `sentence-transformers`）要能降级为关键词/规则召回，保证功能不断。
- **缓存**：对最近问题与召回结果做短期缓存，连续追问时明显提速。

### 5.3 流式输出：把“等待”变成“阅读”
- UI 侧以**增量渲染**呈现 token，首 token 越快越“像实时”。
- 同时要做**节流**：例如每 30–60ms 刷新一次 UI，避免 token 过密导致 UI 卡顿。

---

## 6. UI/交互优化：隐形悬浮窗的“工程细节”

### 6.1 点击穿透与模式切换
- 默认 **click-through**，避免影响面试操作。
- 提供热键切换为**可拖拽/可选中**，并加“强制隐身（全穿透）”兜底热键，避免误操作。

### 6.2 多窗口隔离：语音窗与视觉窗互不拖累
- 视觉链路失败只影响视觉窗；语音链路继续稳定输出。
- 两个 overlay 的交互状态独立管理，避免“切换一个窗口把另一个搞坏”。

---

## 7. 视觉链路优化：截图到答案的延迟与成本控制

### 7.1 图片压缩是最划算的优化
- 截图后转 JPEG 并控制质量/分辨率，能显著降低上传与推理时间。
- 兼顾“可读性”：代码截图要保证字体仍清晰，必要时对长图分块。

### 7.2 异步与超时
- 截图到 Vision LLM 必须异步执行，UI 立即反馈“已捕获/处理中”。
- 设置超时与重试策略，失败时给出可理解的提示（例如网络/限流/无 key）。

---

## 8. 稳定性工程：失败可恢复比“永不失败”更现实

### 8.1 典型故障与对策
- **ASR 断流**：watchdog + 自动重连 + 本地静音补齐。
- **LLM 限流/超时**：指数退避重试 + 降级输出（要点更短）+ 记录失败原因。
- **UI 卡顿**：流式 UI 刷新节流 + 计算/网络放后台线程。
- **依赖缺失（如 PyQt6 DLL）**：给出明确安装路径（conda-forge / VC++ runtime），并保证核心功能能启动到可诊断状态。

### 8.2 隔离原则
- 语音链路与视觉链路**资源隔离、异常隔离、日志隔离**。
- 每条链路都要能独立重启，不需要“重启整个应用”。

---

## 9. 优化落地路线：按收益从大到小做

- **第一优先级（立竿见影）**：分句策略（Partial Segmenter）+ 流式 UI + prompt 裁剪
- **第二优先级（稳定性）**：watchdog、自愈重连、超时/重试、隔离视觉链路
- **第三优先级（成本/体验）**：图片压缩、RAG chunk 与缓存、token 速率与 UI 节流
- **长期项（可观测）**：统一埋点与可回放日志，让每一次“变慢”都能被定位

---

## 10. 结语：实时系统优化的本质是“关键路径减法”
实时面试辅助并不需要“最聪明的模型”，而需要一套对现实世界抖动友好的工程系统：**快**靠切句与流式，**稳**靠隔离与自愈，**准**靠受控的 RAG 与 prompt 预算。把这三件事做好，用户体感会从“工具”变成“副驾驶”。
